# oxlint Migration Steps

This document records all steps taken and remaining to migrate from ESLint to oxlint.

## Current state (as of this document)

The following changes have already been applied to the working tree but not yet committed:

- **`.oxlintrc.json`** — generated by `npx @oxlint/migrate` and adjusted:
  - Added `"oxlint"` to the `no-comments/disallowComments` allow list so
    `// oxlint-disable-next-line` suppression comments are permitted.
- **`eslint.config.mjs`** — added `"oxlint"` to the `no-comments/disallowComments`
  allow list (same reason; ESLint runs in parallel and must allow the same comments).
- **`test/EffectTest.ts`** — added `// oxlint-disable-next-line typescript/no-invalid-void-type`
  to suppress a false positive: oxlint's `no-invalid-void-type` implementation lacks the
  `allowInGenericTypeArguments` option that typescript-eslint defaults to `true`. The code
  is valid TypeScript (instantiation expression passing `void` as a generic type argument).
- **`package.json`** — `oxlint 1.64.0` added as a dev dependency.
- **`Makefile`** — `lint-ts` target updated to run `npx oxlint .` before ESLint.

## Step 2 — Add type-aware rules to oxlint

Re-run the migration tool with the `--type-aware` flag to include the 48 type-aware
`@typescript-eslint/*` rules that require TypeScript type information:

```sh
npx @oxlint/migrate --type-aware
```

Update the Makefile `lint-ts` target to pass `--tsconfig`:

```makefile
lint-ts: node_modules src/manifest.json
	npx oxlint --tsconfig tsconfig.json .
	npx eslint --cache --cache-location ".cache/eslint/" --max-warnings 0
```

Run both linters and fix any new errors or false positives in `.oxlintrc.json` overrides.

## Step 3 — Contract: slim down ESLint to only unmigrated rules

Once oxlint covers everything it can, replace `eslint.config.mjs` with a minimal config
that contains only the rules oxlint cannot run. This removes the duplicate effort and makes
the remaining ESLint scope explicit.

The minimal ESLint config should keep only:

```js
// Only rules that oxlint cannot cover — see oxlint-migration.md for details
import pluginImport from 'eslint-plugin-import'
import { defineConfig, globalIgnores } from 'eslint/config'
import typescriptEslint from 'typescript-eslint'

export default defineConfig([
  globalIgnores(['.cache/', '.dev/', 'assets/locales/', 'data/', 'dist/',
    'integration-results/', 'src/locales/', '*.cjs', '*.mjs']),
  typescriptEslint.configs.strictTypeChecked,      // for type-aware rules not yet in oxlint
  pluginImport.flatConfigs.recommended,
  pluginImport.flatConfigs.typescript,
  {
    linterOptions: { reportUnusedDisableDirectives: 'error' },
    languageOptions: {
      parserOptions: { projectService: true, tsconfigRootDir: import.meta.dirname },
    },
    settings: { 'import/resolver': { typescript: true } },
  },
  {
    rules: {
      'import/no-extraneous-dependencies': ['error', {
        devDependencies: ['assets/**/*.ts', 'integration/**/*.ts',
          'test/**/*.ts', 'visual-regression/**/*.ts', '*.ts'],
      }],
      'import/no-internal-modules': ['error', {
        allow: [
          'fp-ts/lib/!(Array|boolean|Eq|function|Identity|NonEmptyArray|Option|Ord|Ordering|Predicate|ReadonlyArray|ReadonlyNonEmptyArray|ReadonlyRecord|ReadonlySet|ReadonlyTuple|Refinement|string).js',
          'io-ts/lib/*', 'iso-639-3/to-*.json', 'logging-ts/lib/*',
          'remixicon/**/*.svg', 'integration/base', 'types/*',
          'vitest/config', '*/index.ts',
        ],
      }],
      'no-restricted-syntax': ['error',
        { selector: 'ImportDeclaration[specifiers.length = 0]',
          message: 'Empty imports are not allowed' }],
    },
  },
  {
    files: ['src/RefactoringUtilities/**/*.ts'],
    rules: {
      'import/no-internal-modules': ['error', { allow: ['fp-ts/lib/*', 'io-ts/lib/*'] }],
    },
  },
  {
    files: ['integration/**/*.ts', 'test/**/*.ts', 'visual-regression/**/*.ts'],
    rules: {
      'import/no-internal-modules': ['error', { forbid: ['**/assets/**/*'] }],
    },
  },
])
```

## Step 4 — Remove ESLint entirely

If `import/no-internal-modules` is converted to `no-restricted-imports` in oxlint (see
migration notes below), and `import/no-extraneous-dependencies` is either accepted as a
gap or replaced by another tool, ESLint can be removed:

```sh
pnpm remove eslint eslint-config-prettier eslint-import-resolver-typescript \
  eslint-plugin-import eslint-plugin-no-comments eslint-plugin-wc \
  typescript-eslint @eslint/js
```

Update `Makefile`:
```makefile
lint-ts: node_modules src/manifest.json
	npx oxlint --tsconfig tsconfig.json .
```

Update `package.json` scripts:
```json
"lint:ts": "oxlint --tsconfig tsconfig.json ."
```

Delete `eslint.config.mjs`.

Commit message: `contract: remove eslint, oxlint is now the sole ts linter`

## Rules that cannot be migrated

### `import/no-extraneous-dependencies`

**Why it can't be migrated:** Not yet implemented in oxlint
([tracking issue](https://github.com/oxc-project/oxc/issues)).

**Purpose:** Ensures every import comes from a package listed as a dependency (or
devDependency for test/asset files) in `package.json`, preventing accidental reliance on
transitive dependencies.

**Useful for this repo:** Yes — the repo has a clear production/dev boundary and the rule
enforces it.

**Replacement options:**
- Keep ESLint running only for this rule (Step 3 above).
- Use `knip` or `depcheck` as a separate CI step to detect missing/extra dependencies.
- Wait for oxlint to implement the rule.

---

### `no-restricted-syntax` (empty imports)

**Why it can't be migrated:** Not yet implemented in oxlint.

**Purpose:** The config uses `{ selector: 'ImportDeclaration[specifiers.length = 0]' }` to
forbid `import {}` statements that import nothing.

**Useful for this repo:** Yes — empty imports are dead code.

**Replacement options:**
- Keep ESLint for this rule (Step 3).
- Oxlint's `no-unused-expressions` does not cover import statements.
- A Prettier/custom script could strip empty imports at format time.
- File an oxlint feature request for `no-empty-import` or similar.

---

### `import/no-internal-modules`

**Why it can't be migrated:** Explicitly marked "won't be implemented" by the oxlint team.
Their rationale: `no-restricted-imports` (which oxlint *does* support) can cover the same
use-cases.

**Purpose:** Controls which internal subpaths of packages (e.g. `fp-ts/lib/Option`,
`io-ts/lib/*`) may be imported directly, rather than going through a package's public API.

**Useful for this repo:** Yes — the repo deliberately allowlists specific fp-ts/io-ts
internals and forbids others, enforcing a migration boundary.

**Replacement options:**
- Convert the allowlist to `no-restricted-imports` patterns in `.oxlintrc.json`. This
  requires rewriting each `allow` glob as a `patterns` deny entry (inverted logic), which
  is more verbose but feasible.
- Keep ESLint for this rule until the conversion is done.

---

### `quotes`

**Why it can't be migrated:** Marked as a deprecated stylistic rule by the oxlint team;
they recommend using a formatter instead.

**Purpose:** Enforces single-quoted strings.

**Useful for this repo:** Partially — Prettier already enforces single quotes, so this rule
is redundant. The ESLint config applied it on top of `eslint-config-prettier`, which
suggests it was intentional at some point but Prettier covers it now.

**Replacement options:** Rely on Prettier (already configured). No action needed in oxlint.

---

### Type-aware `@typescript-eslint/*` rules (48 rules)

**Why they are skipped by default:** They require the TypeScript compiler's type checker,
which oxlint does not load unless invoked with `--tsconfig`.

**Key rules affected:**
- `@typescript-eslint/prefer-nullish-coalescing` (with `ignoreConditionalTests`)
- `@typescript-eslint/restrict-template-expressions` (with `allowNumber`)
- `@typescript-eslint/strict-boolean-expressions`
- `@typescript-eslint/consistent-type-exports`
- `@typescript-eslint/no-unsafe-argument`, `no-unsafe-assignment`,
  `no-unsafe-member-access`, `no-unsafe-return`
- `@typescript-eslint/unbound-method`, `prefer-promise-reject-errors`, and more

**Useful for this repo:** Yes — these are the strictest safety checks.

**Replacement options:** Re-run `npx @oxlint/migrate --type-aware` and run oxlint with
`npx oxlint --tsconfig tsconfig.json .` (Step 2 above). This is the intended migration path.

---

### `typescript/no-invalid-void-type` (partial — false positive)

**Why it is partially broken:** Oxlint's implementation does not support the
`allowInGenericTypeArguments` option (default `true` in typescript-eslint). It incorrectly
flags `void` used as a generic type argument in TypeScript instantiation expressions.

**Workaround applied:** `// oxlint-disable-next-line typescript/no-invalid-void-type`
added to `test/EffectTest.ts` line 9. The rule remains enabled for all other cases.

**Replacement options:** File an oxlint issue for `allowInGenericTypeArguments` support.
Remove the suppress comment once it is fixed.
